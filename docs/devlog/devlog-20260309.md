# Cloudflare Workers Error 1101 (Worker Hung) 트러블슈팅

> **프로젝트**: 어이어이바위게 (OiOiBWG) — QWER 비공식 응원법 싱크 뷰어  
> **스택**: Next.js 16 (App Router) + vinext + Cloudflare Workers + Hyperdrive + Supabase + Drizzle ORM  
> **증상**: 페이지 접속 시 `Error 1101: Workers runtime canceled this request because it detected that your Worker's code had hung and would never generate a response.`  
> **소요 시간**: 약 4시간의 삽질  
> **결론**: 단 26줄의 코드로 해결

---

## 🔥 증상

`/` (메인) 페이지 접속 시 다음 에러가 발생하며 페이지가 열리지 않음:

```bash
Error 1101
Workers runtime canceled this request because it detected that
your Worker's code had hung and would never generate a response.
```

- 로컬 개발 서버(`pnpm run dev`)에서도 동일하게 발생
- 앨범 리스트를 DB에서 조회하는 [getAllSongs()](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/src/libs/db/drizzle/queries.ts#25-34) 호출 시 301초 대기 후 timeout
- Wrangler tail 로그에 `Promise will never complete` 스택트레이스 출력
- `/albums/manito` (앨범 상세 페이지)는 정상적으로 열림 → DB 쿼리 자체의 문제가 아님

## 🧪 시도했던 (잘못된) 접근들

### ❌ 시도 1: `idle_timeout: 0` 설정

`postgres.js`의 `idle_timeout`을 0으로 설정하면 소켓을 즉시 반환하니까 해결될 거라 생각했다.

```ts
const queryClient = postgres(connectionString, {
  prepare: false,
  idle_timeout: 0, // ← 이게 해결책이라 믿었다
});
```

**결과**: 여전히 동일하게 Hung 발생. 근본 원인이 아니었음.

---

### ❌ 시도 2: [pg](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#3472-3473) (node-postgres) 드라이버로 교체

`postgres.js` 대신 [pg](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#3472-3473)를 쓰면 커넥션 관리가 더 낫지 않을까 생각했다. `Pool`에 `allowExitOnIdle: true`까지 추가.

```ts
import { Pool } from "pg";

const pool = new Pool({
  connectionString,
  allowExitOnIdle: true, // ← 이게 핵심이라 믿었다
  max: 1,
});
```

**결과**: Cloudflare Workers 환경에서 [pg](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#3472-3473)는 네이티브 Node.js 모듈 의존성이 강해서 오히려 빌드 에러 및 더 심한 Hung 발생. [Hyperdrive](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#10912-10959) 위에 `Pool`을 또 올리는 이중 풀링이 되어 최악.

---

### ❌ 시도 3: `globalThis` 싱글톤 패턴

HMR 재시작 시 연결이 중복 생성되지 않도록 전역 싱글톤으로 관리하려 했다.

```ts
const globalForDb = globalThis as unknown as { conn: postgres.Sql | undefined };

export function getDb() {
  if (globalForDb.conn) return drizzle(globalForDb.conn, { schema });
  globalForDb.conn = queryClient;
  return drizzle(queryClient, { schema });
}
```

**결과**: Cloudflare Workers에서는 전역 변수에 TCP 소켓이 살아있으면 응답 후에도 소켓 때문에 정지하지 못하고, 런타임이 "Hung"으로 판단해 강제 종료함. 서버 환경에서는 완벽한 패턴이지만 **엣지 런타임에서는 최악의 안티패턴**.

---

### ❌ 시도 4: `next/server`의 [after()](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#1701-1702) 훅으로 커넥션 종료

요청이 끝난 후 [after()](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#1701-1702)로 `queryClient.end()`를 호출해 TCP 소켓을 닫으려 했다.

```ts
import { after } from "next/server";
import { cache } from "react";

export const getDb = cache(async () => {
  after(async () => {
    await queryClient.end(); // ← 요청 후 소켓 강제 종료
  });
  return drizzle(queryClient, { schema });
});
```

**결과**: [after()](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#1701-1702) 자체는 올바른 아이디어였지만, 아직 근본 원인(`connectionString`이 undefined)을 해결하지 못한 상태에서 적용했기 때문에 의미 없었음.

---

### ❌ 시도 5: Worker Entry에서 커스텀 헤더로 connectionString 주입

`env.DB.connectionString`을 [worker/index.ts](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker/index.ts)에서 커스텀 헤더로 주입하고, `db/index.ts`에서 `next/headers`로 꺼내 쓰는 방식.

```ts
// worker/index.ts
const modifiedRequest = new Request(request, {
  headers: {
    ...Object.fromEntries(request.headers.entries()),
    "x-hyperdrive-connection-string": env.DB?.connectionString, // ← DB 비밀번호가 헤더에 노출!
  },
});
return handler.fetch(modifiedRequest);
```

**결과**: vinext 공식 문서에서 명시적으로 금지하는 안티패턴. 불필요하게 복잡하고, 헤더에 DB 연결 문자열(비밀번호 포함)을 노출하는 보안 취약점.

## ✅ 최종 해결책

### 핵심 원인: `connectionString`이 `undefined`였다

Cloudflare Workers에서 `env.DB`(Hyperdrive 바인딩)는 `process.env`가 아니라 **Worker의 [env](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/.env) 객체**에 있다. 그런데 우리 코드는 계속 이걸 `process.env.DATABASE_URL`로 접근하려 하거나, 잘못된 방법으로 가져오려고 했다.

### 해결 방법: `import { env } from "cloudflare:workers"`

vinext 공식 문서와 Cloudflare Workers 공식 문서에 명시된 방법:

```ts
// src/libs/db/drizzle/index.ts — 최종 버전 (단 26줄)
import { env } from "cloudflare:workers";
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import { cache } from "react";

import * as schema from "./schema";

export const getDb = cache(() => {
  const connectionString = (env.DB as Hyperdrive)?.connectionString || process.env.DATABASE_URL!; // 프로덕션: Hyperdrive 프록시 주소 // 로컬: Miniflare가 주입한 직접 주소

  const queryClient = postgres(connectionString, {
    prepare: false, // Hyperdrive/PgBouncer 필수 설정
    max: 5, // Cloudflare 공식 권장값
    fetch_types: false, // 불필요한 라운드트립 제거
  });

  return drizzle(queryClient, { schema });
});
```

**이것만으로 해결됐다.**

### 왜 이게 동작하는가?

- `@cloudflare/vite-plugin`이 RSC(서버 컴포넌트) 환경을 실제 `workerd` 런타임에서 실행하기 때문에, `cloudflare:workers`가 네이티브 모듈로 바로 작동한다.
- 로컬 개발 시에는 Miniflare가 Hyperdrive를 에뮬레이트하고, [.env](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/.env)의 `CLOUDFLARE_HYPERDRIVE_LOCAL_CONNECTION_STRING_DB` 값을 자동으로 주입해준다.
- 별도로 [worker/index.ts](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker/index.ts)를 수정하거나, `getPlatformProxy()`, `getRequestContext()` 같은 우회책이 **전혀 필요 없다**.

## 🚫 vinext에서 절대 하지 말아야 할 것들

| 안티패턴                                                                                                                | 이유                                               |
| ----------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| `globalThis`에 DB 커넥션 캐싱                                                                                           | TCP 소켓이 살아있어 Worker가 Hung 상태로 판단됨    |
| [pg](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/worker-configuration.d.ts#3472-3473) (node-postgres) 사용 | 네이티브 Node.js 의존성이 강해 엣지 환경 빌드 실패 |
| Hyperdrive 위에 `Pool` 추가                                                                                             | 이중 풀링 → 성능 낭비 + Hung 가능성 증가           |
| Worker Entry에서 [env](file:///home/hsj95/workspace/dev/project/cheer-rock-crab/.env)를 헤더로 전달                     | 불필요한 복잡성 + 비밀번호 노출                    |
| `getPlatformProxy()` / `getRequestContext()`                                                                            | 구식 패턴, `cloudflare:workers` 사용할 것          |
| `"use cache"` 지시어를 DB 쿼리 함수에 추가                                                                              | Miniflare + postgres.js 환경에서 Hung 유발         |

## 💡 핵심 교훈

1. **공식 문서와 skill 파일을 먼저 읽어라.** SKILL.md에 모든 답이 있었다. 읽지 않고 직관으로 추측하면 4시간이 날아간다.
2. **엣지 런타임의 라이프사이클은 일반 서버와 다르다.** 서버에서 완벽한 패턴(싱글톤, Pool)이 엣지에서는 최악의 안티패턴이 될 수 있다.
3. **Hyperdrive가 이미 커넥션 풀링을 한다.** 워커 내부에서 Pool을 또 만들 필요 없다.
4. **`connectionString`이 `undefined`인지 먼저 확인하라.** 아무리 설정을 잘 해도 연결 주소가 없으면 아무것도 안 된다.
5. **에러 메시지를 믿어라.** `Promise will never complete`는 말 그대로 소켓이 영원히 기다리고 있다는 뜻이다.
