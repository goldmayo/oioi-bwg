# Next.js 16 렌더링 전략 분석 및 리뉴얼 피드백

## 1. 개요

본 문서는 현재 프로젝트의 페이지별 렌더링 전략(SSR, SSG, ISR, CSR)을 스캔한 뒤, 성능(로딩 속도 향상, DB/서버 병목 방지)을 극대화하기 위한 Next.js 16 (React 19) 기반의 개선안을 정리한 문서입니다.

## 2. 현재 상태 (AS-IS) 렌더링 전략 분석

### 2.1 `/src/app/(user)/page.tsx` (메인 홈 페이지)

- **현재 전략**: SSR (Server-Side Rendering)
- **분석 요약**: `getAllSongs()`를 호출하여 DB에서 모든 곡 메타데이터를 직접 조회합니다. 데이터 캐싱(`use cache`나 `unstable_cache`) 처리가 없어 유저 접속 시마다 렌더링 및 DB 쿼리가 산발적으로 발생합니다.

### 2.2 `/src/app/(user)/songs/[slug]/page.tsx` (가사 상세 뷰어)

- **현재 전략**: SSR (Server-Side Rendering)
- **분석 요약**: URL의 `[slug]` 파라미터를 기반으로 `getSongBySlug(slug)` 단건 조회가 실행됩니다. 캐싱 없이 실시간으로 조회하므로, 콘서트 등 트래픽이 몰릴 때 DB에 과부하를 초래할 수 있습니다.

### 2.3 `/src/app/(user)/albums/[slug]/page.tsx` (앨범 상세 페이지)

- **현재 전략**: Cached SSR (SSG/ISR 효과 발휘)
- **분석 요약**: Next.js 16의 실험적 기능인 `"use cache"` 지시어를 컴포넌트 최상단에 사용하여, 하드코딩된 정적 상수(`ALBUMS`) 데이터를 매우 빠르고 쾌적하게 렌더링하여 반환하고 있습니다.

### 2.4 `/src/app/(admin)/...` (관리자 페이지 및 에디터)

- **현재 전략**: SSR + CSR 결합
- **분석 요약**: 실시간 데이터 무결성이 생명인 어드민에서는 SSR을 통해 캐시 없는 DB 원본 데이터를 안전하게 내려받고, 편집기 등 인터랙션 요소는 CSR로 처리하는 베스트 프랙티스(Best Practice)를 따르고 있습니다.

## 3. 리뉴얼 개선 및 최적화 피드백 (TO-BE)

이번 `v0.2.1` 리뉴얼 및 MVP 유저 트래픽(모바일 75%의 다수 접속자)을 고려하여 도입해야 할 주요 전략입니다.

### 3.1 💡 메인 홈 및 디스코그래피 목록 (`/page.tsx`)

- **목표 렌더링 전략**: **On-Demand ISR (주문형 재검증) 또는 SSG**
- **적용 이유**: 앨범 목록이나 곡 정보는 '관리자의 데이터 편집 시'에만 드물게 변경됩니다. 모든 방문자에게 100% 동일한 화면을 보여주는 지면이므로 런타임에 DB를 찌를 이유가 없습니다.
- **적용 방법**: 페이지 로직이나 데이터 패칭 함수 상단에 `"use cache"`를 적용해 무기한 캐싱합니다. 관리자 메뉴에서 새 곡이나 앨범 등록 시(Server Action 단계에서) `revalidatePath("/")`를 트리거하여 전체 리스트 화면 캐시를 갱신합니다.

### 3.2 💡 가사 & 응원법 상세 뷰어 (`/songs/[slug]/page.tsx`)

- **목표 렌더링 전략**: **곡 단위 태그 기반의 On-Demand ISR (가장 중요)**
- **적용 이유**: 75% 이상의 모바일 유저들이 현장이나 이벤트 전 특정 곡 뷰어에 막대한 트래픽을 단기적으로 발생시킬 확률이 높습니다. Cloudflare Workers 위에서 동작 중이므로 Edge 단까지 뻗는 강력한 캐시 활용이 필수적입니다.
- **적용 방법**:
  - `unstable_cache` 또는 `"use cache"` 지시어에 기반을 두어 데이터를 가져올 때 `tags: ['song-[slug]']` 속성으로 곡별 키를 부착합니다.
  - 이렇게 구성하면 사용자는 DB 호출 횟수 '0'에 수렴하는 속도(마치 정적 페이지)로 가사 페이지를 보게 됩니다.
  - 관리자 편집기(`/admin/edit`)에서 가사를 **저장**하는 순간, `revalidateTag('song-해당슬러그')`를 던져 수정된 곡의 응원법 페이지만 갱신시킵니다.

### 3.3 💡 앨범 상세 목록 페이지 (`/albums/[slug]/page.tsx`)

- **목표 렌더링 전략**: **`generateStaticParams`를 활용한 완벽한 100% SSG 구축**
- **적용 이유**: 이미 `"use cache"`가 발려 있어서 응답 성능은 최상이지만, 서비스 특성상 앨범의 갯수(풀)가 한정적이기 때문에 런타임에 첫 페이지 생성(Miss) 과정을 아예 제거하는 것이 좋습니다.
- **적용 방법**: 파일 내에 `export async function generateStaticParams()`를 선언해 현재 존재하는 `ALBUMS`의 슬러그 배열을 전부 반환, 앱 빌드(배포) 과정에서 애초부터 모든 접속 정적 HTML이 구워지도록 합니다.

### 3.4 💡 어드민 관련 페이지

- **목표 렌더링 전략**: **현행 유지 (순수 SSR + CSR 조합)**
- **적용 이유**: 절대 캐시가 입혀져서는 안되는 영역입니다. 데이터 정합성이 필수이므로 런타임에 DB 데이터를 그대로 요청해 들고 오도록 둡니다.
